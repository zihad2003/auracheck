## 1. Executive Summary

The AuraCheck weather telemetry application was audited across three dimensions — security, performance, and future UI direction — by a team of specialized agents working in parallel with cross-review verification.

### Security (18 findings: 2 Critical · 5 High · 6 Medium · 5 Low)

The backend has **no authentication on any endpoint**, including write routes for ground truth data and a telemetry configuration route that enables Server-Side Request Forgery (SSRF). User-supplied query parameters are passed directly to Mongoose queries without sanitization, creating **NoSQL injection** vulnerabilities across multiple routes. Internal error details are leaked to clients on every error. The `joi` validation library is installed but never used — a missed opportunity that partially caused these gaps.

### Performance (16 findings: 4 High · 6 Medium · 6 Low)

The server runs **two redundant `setInterval` loops** generating and broadcasting telemetry every 2 seconds, even with zero connected clients. Weather data is fetched from the OpenWeatherMap API with **no caching**, causing duplicate external API calls on every user interaction. The frontend has ~300KB of **unused dependencies** inflating the bundle, no code splitting across pages, and React hooks that poll redundantly alongside an active WebSocket.

### 3D UI Proposal

Creative-Developer designed a **weather-reactive 3D interface** using React Three Fiber, Drei, and framer-motion-3d — compatible with the existing Vite 5 / React 18 / TypeScript / Tailwind stack with no configuration changes. The proposal covers 7 component mappings, a full weather atmosphere system (wind particles, precipitation, temperature-driven lighting, storm effects), and a 5-phase gradual integration plan that keeps the flat UI functional at every stage.

### Cross-Review Verdict

The 3D proposal is **sound in architecture but requires hardening in execution**. Performance review found 4 high-severity issues in the code samples (incorrect particle update pattern, unbounded WebGL render loop, bundle impact not honestly accounted for, simultaneous frame-budget overcommitment during storms). Security review found the proposal safe from injection vectors as designed, but identified XSS risk if `dangerouslySetInnerHTML` is ever used in Drei `<Html>` portals, and flagged supply chain exposure from new dependencies.

**Recommendation: Fix Critical/High backend vulnerabilities first. Then implement the 3D proposal with the corrections identified in the cross-review.**

## 2. Security Findings

*Audited by: Security-Auditor*
*Files covered: all backend routes, models, utils, .env files, frontend weatherApi.ts*

### 2.1 Critical

#### SEC-01 · NoSQL Injection via Unsanitized Query Parameters
- **Files**: `backend/routes/activity.js:109-114`, `backend/routes/groundTruth.js:73`, `backend/routes/activity.js:149`
- **Issue**: `userId`, `activity`, `activityId`, and `areaCode` are passed directly from `req.params` / `req.body` into Mongoose `find()`, `findOneAndUpdate()`, and `countDocuments()` without sanitization. An attacker can send a JSON object (e.g., `{"userId": {"$gt": ""}}`) to manipulate query logic. The `joi` library is installed but never imported or used in any route.
- **Fix**: Install `mongo-sanitize` and call `sanitize()` on all route inputs. For ObjectId fields, validate with `mongoose.Types.ObjectId.isValid()`. Implement Joi validation schemas for all routes (the dependency is already there — use it).

#### SEC-02 · Unauthenticated Telemetry Configuration Endpoint Enables SSRF
- **File**: `backend/routes/telemetry.js:49-68`
- **Issue**: `POST /api/telemetry/configure` accepts `apiEndpoint`, `apiKey`, and `sourceId` from any unauthenticated caller. The `fetchExternalTelemetry` method in `telemetryService.js:55-73` will then make HTTP requests to that attacker-controlled URL, enabling SSRF against internal services (cloud metadata endpoints, internal network).
- **Fix**: Add authentication middleware. Validate `apiEndpoint` against an allowlist. Block requests to private IP ranges in `fetchExternalTelemetry`.

---

### 2.2 High

#### SEC-03 · No Authentication or Authorization on Any Route
- **Files**: `backend/server.js:50-54`, all route files
- **Issue**: The entire API is unauthenticated. All write endpoints (`POST /api/ground-truth`, `POST /api/activity/score`, `POST /api/activity/feedback`, `POST /api/telemetry/configure`) are open to anyone. The `userId` field is self-reported with no verification.
- **Fix**: Implement JWT or session-based authentication. Guard all write endpoints. Ensure users can only modify their own data.

#### SEC-04 · Socket.IO Room Injection via Unsanitized `areaCode`
- **File**: `backend/server.js:68-71`
- **Issue**: The `join-area` socket event calls `socket.join(areaCode)` with a client-supplied string, with no validation. An attacker could join arbitrary rooms or inject special characters.
- **Fix**: Validate `areaCode` against an expected format (e.g., `/^\d{4,6}$/`) before calling `socket.join()`.

#### SEC-05 · Error Messages Leak Internal Details
- **Files**: `backend/routes/activity.js:77-78`, `backend/routes/groundTruth.js:63-64`, `backend/routes/skyScan.js:26-27`, `backend/routes/weather.js:35-36`
- **Issue**: All catch blocks return `error.message` to the client, leaking Mongoose errors, stack traces, file paths, and connection details.
- **Fix**: Return only a generic error message to clients. Log the full error server-side. Add a centralized error handler in `server.js`.

#### SEC-06 · Unbounded Request Body Size
- **File**: `backend/server.js:41`
- **Issue**: `express.json()` has no `limit` set. The sky-scan route accepts image data in the JSON body — a large base64 payload can exhaust server memory.
- **Fix**: `app.use(express.json({ limit: '100kb' }))`. Apply a higher limit specifically to the sky-scan route if needed.

#### SEC-07 · SSRF in `TelemetryService.fetchExternalTelemetry`
- **File**: `backend/utils/telemetryService.js:55-73`
- **Issue**: Makes HTTP requests to any `apiEndpoint` URL. Combined with the unauthenticated configure endpoint (SEC-02), this is a direct SSRF path.
- **Fix**: URL allowlist + private IP range blocking (see SEC-02).

---

### 2.3 Medium

#### SEC-08 · Unbounded `limit`, `hours`, and `radius` Query Parameters
- **Files**: `backend/routes/activity.js:107`, `backend/routes/groundTruth.js:71,95,139`
- **Issue**: `parseInt(limit)` with no upper bound allows `limit=999999999`, forcing enormous DB result sets.
- **Fix**: `const safeLimit = Math.min(parseInt(limit) || 20, 100)`. Apply equivalent bounds to `hours` (max 168) and `radius` (max 50km).

#### SEC-09 · `req.io` Not Attached — Runtime Crash on Ground Truth POST
- **File**: `backend/routes/groundTruth.js:48`
- **Issue**: Calls `req.io.to(areaCode).emit(...)` but no middleware attaches `io` to `req`. Every POST to `/api/ground-truth` crashes with `TypeError`. The report is saved before the crash — partial state persists.
- **Fix**: Add `app.use((req, res, next) => { req.io = io; next(); });` in `server.js` before route mounts.

#### SEC-10 · Prototype Pollution Risk in `aggregateTelemetry`
- **File**: `backend/utils/telemetryService.js:85-90`
- **Issue**: `Object.assign(aggregated.motion, data.motion)` merges from potentially external-sourced telemetry data. If `data.motion` contains `__proto__` or `constructor`, prototype pollution occurs.
- **Fix**: Use a safe merge that strips `__proto__` / `constructor` keys, or validate the telemetry data shape on ingestion.

#### SEC-11 · Frontend Exposes API Keys via `VITE_` Prefix
- **File**: `frontend/.env:1-2`
- **Issue**: `VITE_OPENWEATHER_API_KEY` and `VITE_MAPBOX_ACCESS_TOKEN` are embedded in the client bundle at build time and visible to any user.
- **Fix**: Proxy all weather API requests through the backend. Remove these keys from the frontend. Apply domain restrictions to the Mapbox token.

#### SEC-12 · CORS Fallback May Allow Misconfiguration in Production
- **Files**: `backend/server.js:22-24,37-40`
- **Issue**: Falls back to `http://localhost:3000` when `FRONTEND_URL` is unset. If `FRONTEND_URL` is set to `*` by mistake, credentials are exposed to any origin.
- **Fix**: Validate `FRONTEND_URL` is set at startup when `NODE_ENV=production`. Consider an array allowlist over a single origin string.

#### SEC-13 · `joi` Installed but Never Used
- **File**: `backend/package.json:20`
- **Issue**: The validation library is present but no route uses it. This is the root cause of several injection findings above.
- **Fix**: Implement Joi validation schemas for all route inputs (fixes SEC-01, SEC-08, and several others simultaneously).

## 3. Performance Findings

*Audited by: Performance-Engineer*
*Files covered: all backend routes, utils, frontend hooks, components, pages*

### 3.1 High

#### PERF-01 · No API Response Caching for Weather Data
- **File**: `backend/utils/weatherService.js:10-35`
- **Issue**: `getOneCallWeather()` and `getCurrentWeather()` make fresh HTTP requests to OpenWeatherMap on every call. Weather data changes on a minutes timescale; the app fetches it on every user request.
- **Fix**: Add an in-memory cache (e.g., `node-cache`) keyed by rounded lat/lon with a 60-120 second TTL.
  ```js
  const key = `${lat.toFixed(2)}_${lon.toFixed(2)}`;
  const cached = cache.get(key);
  if (cached) return cached;
  ```

#### PERF-02 · Duplicate Weather API Call on Every Activity Score Request
- **Files**: `frontend/src/pages/ActivityDashboard.tsx:69-90,92-114`, `backend/routes/activity.js:32`
- **Issue**: The frontend calls both `GET /api/weather/:lat/:lon` and `POST /api/activity/score`. The score endpoint internally calls `weatherService.getOneCallWeather()` again — 2 identical external API fetches per user interaction.
- **Fix**: (a) Pass already-fetched weather data to the score endpoint, or (b) implement PERF-01 caching, or (c) merge the two endpoints.

#### PERF-03 · Cascading `useEffect` Re-fetches with No Debounce
- **File**: `frontend/src/pages/ActivityDashboard.tsx:44-48`
- **Issue**: The `useEffect` fires `calculateScore()` on every `selectedActivity`, `weatherData`, or `userLocation` change — including when `fetchWeatherData()` sets `weatherData`, triggering another score POST. Rapid activity switching queues concurrent requests with no cancellation.
- **Fix**: (a) Debounce score calculation by 300-500ms. (b) Remove `userLocation` from the dependency array. (c) Add `AbortController` to cancel in-flight requests.

#### PERF-04 · Dual Telemetry `setInterval` Running Unconditionally
- **Files**: `backend/utils/telemetryService.js:42-48`, `backend/server.js:85-94`
- **Issue**: `TelemetryService.startMockDataStream()` generates telemetry every 2s. `server.js` runs a second `setInterval` every 2s to broadcast via Socket.IO. Both run 24/7 even with zero connected clients. Neither interval has a shutdown path — graceful process exit is impossible.
- **Fix**: (a) Remove `startMockDataStream()` — generate data on-demand in the broadcast interval only. (b) Guard broadcast: `if (io.sockets.adapter.rooms.get('telemetry-stream')?.size > 0)`. (c) Store interval IDs and clear them on `SIGTERM`.

## 4. 3D UI Proposal

*Designed by: Creative-Developer*
*Research basis: React Three Fiber v8 + Drei v9 documentation (current APIs)*

### 4.1 Stack Decision

**React Three Fiber (@react-three/fiber v8) + Drei (@react-three/drei v9)** is the correct integration path for AuraCheck's Vite 5 / React 18 / TypeScript / Tailwind stack:

- R3F is a React reconciler for Three.js — scenes are JSX trees, state is managed by hooks, and `useState`, `useEffect`, `useContext`, and custom hooks like `useRealtimeTelemetry` work identically inside and outside the canvas. Zero paradigm shift for the existing codebase.
- Drei provides production-ready abstractions (`MeshTransmissionMaterial`, `Float`, `Html`, `Sky`, `Stars`, `Cloud`, `Text`, `Instances`, `ContactShadows`, `AdaptiveDpr`) that would each require hundreds of lines of raw Three.js boilerplate.
- `framer-motion-3d` extends the existing `framer-motion ^10` dependency to 3D meshes — animation knowledge transfers directly.
- The existing `tsconfig.json` (target ES2020, strict mode, bundler resolution) requires zero changes. `vite.config.ts` requires no changes.

### 4.3 Scene Architecture

```tsx
<Canvas
  camera={{ position: [0, 2, 12], fov: 45, near: 0.1, far: 1000 }}
  dpr={[1, 2]}
  gl={{ antialias: true, alpha: true, powerPreference: 'high-performance' }}
  shadows
  frameloop="demand"  // Cross-review correction: only render when needed
  style={{ position: 'fixed', inset: 0, zIndex: 0 }}
>
  <Suspense fallback={null}>
    <WeatherScene />
  </Suspense>
</Canvas>
```

**Camera**: PerspectiveCamera, FOV 45°, position `[0, 2, 12]`. `OrbitControls` with `enableZoom={false}`, `enablePan={false}`, `autoRotate`, constrained polar angle — gentle ambient rotation, no user disorientation.

**Lighting rig** (weather-reactive):
- `<ambientLight>` — intensity based on time of day (0.15 night / 0.40 day)
- `<directionalLight>` — simulates sun/moon; intensity modulated by cloud cover; `shadow-mapSize={[512, 512]}` or replaced by `<ContactShadows>` (see cross-review XREV-09)
- Rim point light at `[-5, 3, -5]` for panel edge highlights
- Conditional `<LightningFlash>` during storms

**Skybox**: Drei's `<Sky>` + `<Stars>` + `<Cloud>` driven by weather conditions. Smooth transitions via `useSpring` when `weather.current.weather.main` changes.

### 4.5 Weather-Reactive Atmosphere

**Wind** → `WindParticles` component: 2000 particles in a `<Points>` geometry. Position updates in `useFrame` via direct `Float32Array` access (see cross-review correction XREV-01).

**Rain** → `RainSystem`: `<Instances limit={5000}>` instanced cylinders falling downward, density driven by `precipitation.probability`. Splash pool: 50 pre-allocated `InstancedMesh` slots updated in `useFrame` — never mount/unmount React components for transient effects (see XREV-02).

**Temperature** → `useColorTemperature(temp: number)`: Maps Celsius range to Three.js color (freezing blue → mild white → hot orange). Applied to ambient + directional light. Transitions via `THREE.Color.lerp()`.

**Storm** → `StormAtmosphere`: Dense `<fog>`, darkened sky, and irregular lightning implemented as chained `setTimeout` (not `setInterval`) with an `active` guard flag to prevent cleanup leaks (see cross-review correction XREV-06).

**Snow** → `SnowSystem`: Slower fall + sinusoidal horizontal drift; round particles; mutually exclusive with rain (see XREV-10).

**Clear/Sunny** → `<Sky>` with high rayleigh scattering + `<GodRays>` postprocessing + `<Sparkles>` atmospheric haze.

## 5. Cross-Review

*Security-Auditor and Performance-Engineer reviewed the Three.js proposal independently.*

### 5.1 Performance Cross-Review

#### XREV-01 · Wind Particle Loop Uses Slow `getX`/`setX` Pattern [HIGH]
The proposed `WindParticles.useFrame` calls `positions.getX(i)` / `positions.setX(i)` — 8000+ method dispatch calls per frame at 2000 particles × 60fps = 480,000 calls/second. V8 cannot optimize this as a tight numeric loop.

**Required fix:**
```js
const arr = ref.current.geometry.attributes.position.array as Float32Array;
const dx = Math.sin(dirRad) * speed;
const dz = Math.cos(dirRad) * speed;
for (let i = 0; i < count; i++) {
  arr[i * 3]     += dx;
  arr[i * 3 + 2] += dz;
  if (Math.abs(arr[i * 3])     > 15) arr[i * 3]     *= -1;
  if (Math.abs(arr[i * 3 + 2]) > 15) arr[i * 3 + 2] *= -1;
}
ref.current.geometry.attributes.position.needsUpdate = true;
```

#### XREV-02 · Rain Splash Rings Must Be Instanced [HIGH]
The proposal describes a pooled splash system but shows no implementation. If each splash is a React component, 50 mount/unmount cycles per second + 50 draw calls will tank performance. **Required fix:** pre-allocate a single `InstancedMesh` of 50 slots, cycle round-robin in `useFrame`, set scale to zero when inactive.

#### XREV-03 · Bundle Impact: +550-630KB Gzipped, Not Offset by Dead Dep Removal [HIGH]
Dead dependency removal (~300KB) should happen independently and is not a credit against the Three.js addition. The canvas is mounted globally in `App.tsx`, preventing lazy loading of core Three.js code. **Required fix:** (a) Use `frameloop="demand"` globally. (b) Do not count dead dep cleanup as an offset. (c) Consider mounting the canvas only on pages that use it for 3D content heavier than the sky background.

#### XREV-04 · Frame Budget Overcommitment During Storms [HIGH]
In a storm: wind particles (2000 iter/frame) + rain (5000 iter/frame) + telemetry hologram + snow + score gauge + Float components + OrbitControls + R3F overhead + shadow map + postprocessing = **estimated 12-18ms total on a mid-range GPU (borderline 60fps; likely 30-45fps on integrated graphics)**. **Required fix:** (a) Profile with `<Perf>` during development. (b) Reduce wind particle count to 500 when rain is also active. (c) Gate postprocessing behind a GPU tier check. (d) Consider GPGPU for particle physics.

#### XREV-05 · `MeshTransmissionMaterial` Doubles Scene Render Per Panel [MEDIUM]
Renders the entire scene from the material's perspective into an FBO for refraction. Multiple panels = multiple scene re-renders. **Required fix:** Use `MeshPhysicalMaterial` with `transmission` for all but one hero panel; set `samples={1}` and `resolution={256}` on any `MeshTransmissionMaterial` instance.

#### XREV-06 · Storm `setInterval` Has Fixed Random Interval + Leaked `setTimeout` [MEDIUM]
The interval spacing is randomized once at creation (not per tick), and the inner `setTimeout` is not cleared in the cleanup — a state update fires after storm ends. **Required fix:** Use chained `setTimeout` with an `active` flag; or drive `pointLight.intensity` directly via `useRef` in `useFrame` to bypass React reconciliation entirely.

#### XREV-07 · Pre-allocation Promised but Not Demonstrated [MEDIUM]
Section 8.3 says "pre-allocate Vector3, Color objects in refs" but most code samples do not show this. **Required standard for all `useFrame` callbacks:**
```js
const _vec = useRef(new THREE.Vector3());
const _col = useRef(new THREE.Color());
```

#### XREV-08 · Canvas Always Mounted → Perpetual RAF Loop on Every Page [MEDIUM]
`<AuraScene>` in `App.tsx` runs `requestAnimationFrame` continuously on all pages including text-heavy ones. **Required fix:** Add `frameloop="demand"` on `<Canvas>` and use `invalidate()` from `useThree()` only when data changes or on pages with active animation.

#### XREV-09 · Shadow Map and `ContactShadows` Both Enabled [LOW]
The proposal enables both `castShadow` on the directional light and `<ContactShadows>` — redundant. **Fix:** Use only `<ContactShadows>`, which is cheaper for floating UI panels. If shadow mapping is kept, reduce to `512×512`.

#### XREV-10 · Stacked Particle Systems Can Reach 12K+ Particles [LOW]
Night + rain + wind = 5000 (stars) + 5000 (rain) + 2000 (wind) + 50 (splashes) = 12,050 particles. **Fix:** Set a global particle budget cap (~8,000 total); make snow and rain mutually exclusive.

### 5.2 Security Cross-Review

#### XREV-SEC-01 · Supply Chain: 5 New Dependencies Double Frontend Attack Surface [MEDIUM]
`three`, `@react-three/fiber`, `@react-three/drei`, `@react-three/postprocessing`, `framer-motion-3d` each bring transitive dependencies. Drei in particular has a deep tree (troika-three-text, camera-controls, maath, etc.).
**Fix:** Enable `npm audit` in CI. Consider Socket.dev or Snyk for continuous monitoring. Pin exact versions.

#### XREV-SEC-02 · XSS Risk via Drei `<Html>` Portals [HIGH]
`<Html>` renders DOM portals from inside the 3D scene. The proposal uses these for SkyScan analysis results, WeatherMap report sidebar, ActivityDashboard tooltips, and nav icons. API response strings (weather descriptions, ground truth `reportType`, telemetry recommendations) flow into these portals. React's JSX auto-escaping protects against HTML injection when using standard `{variable}` interpolation — but if `dangerouslySetInnerHTML` is ever introduced, this becomes a direct XSS path.
**Fix:** Add ESLint rule `react/no-danger` to ban `dangerouslySetInnerHTML` project-wide, especially in `src/components/three/`. Document as a security invariant.

#### XREV-SEC-03 · `framer-motion-3d@11` / `framer-motion@10` Version Mismatch [LOW]
The existing codebase uses `framer-motion ^10.16.16`. The proposal adds `framer-motion-3d@11`. These must be compatible. **Fix:** Verify compatibility; if not compatible, upgrade both to `^11` together.

#### XREV-SEC-04 · Cross-Origin Texture Loading If Asset URLs Become Data-Driven [MEDIUM]
The proposal mentions `<Environment>` (IBL/skybox). If HDR or texture URLs are ever derived from API responses or user input, `THREE.TextureLoader` will make cross-origin requests to attacker-controlled URLs, enabling IP leakage or malicious image exploit. **Fix:** All Three.js asset URLs must be static imports or same-origin paths. Document as a security invariant: "API response strings must NEVER be used as Three.js asset URLs."

#### XREV-SEC-05 · No `webglcontextlost` Handler [LOW]
GPU context loss (driver crash, OOM) leaves a broken canvas with no fallback. **Fix:** Listen for `webglcontextlost` on the canvas and fall back to the flat 2D UI.

#### XREV-SEC-06 · Particle Count Not Clamped from API Data [LOW]
If `precipitation.probability` from a compromised or buggy API returns a value > 100, and particle count is derived unclamped, thousands of extra instances could crash the browser tab. **Fix:** `Math.min(Math.max(probability, 0), 100)` before mapping to instance count. Global particle budget cap.

#### XREV-SEC-07 · Security Invariant: No API Strings as Three.js Parameters [MEDIUM]
The proposal's architecture correctly avoids using API strings as shader code, asset URLs, or constructor arguments. **Fix:** Formalize this as a team coding rule. Review any future feature additions against this invariant.
